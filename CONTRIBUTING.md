# 🤝 Contributing to DROS

Thank you for your interest in Deep Reasoning OS.

DROS operates as **open architecture, closed execution** — the system design, research methodology, and documentation are public, while the production trading infrastructure remains private.

---

## What You Can Contribute

### ✅ Welcome Contributions

- **Documentation improvements** — clarity, typos, structure, diagrams
- **Architecture discussions** — open an Issue to discuss design questions or trade-offs
- **Research references** — suggest relevant papers on market microstructure, online learning, or safety
- **Correction of technical inaccuracies** — if something in the docs is factually wrong, please flag it
- **FAQ additions** — common questions that should be answered publicly

### ❌ Out of Scope

This is a public documentation repository. The following are not part of this repo and cannot be contributed:

- Trading strategies or parameter tuning suggestions
- Execution code modifications
- ML model changes or weight submissions
- Exchange API integrations

If you have a research partnership or licensing discussion in mind, see [Enterprise & Partnerships](./README.md#-enterprise--partnerships).

---

## How to Contribute

### For Documentation Issues
1. Open a GitHub Issue with label `documentation`
2. Describe what is unclear, incorrect, or missing
3. Submit a Pull Request with the improvement

### For Architecture Discussions
1. Open a GitHub Issue with label `discussion`
2. Describe the question or trade-off you want to explore
3. Keep discussion focused on publicly documented components

### For Research References
1. Open a GitHub Issue with label `research`
2. Include: paper title, authors, DOI/URL, and why it is relevant to DROS
3. Relevant papers may be added to the academic references section

---

## Code of Conduct

- Be respectful and constructive
- Stay within the documented public scope
- No solicitation, spam, or off-topic financial discussion
- Issues about live trading results or performance will be closed without response

---

## Response Times

This is a research project maintained by a small team. Response times are best-effort:

- Documentation fixes: typically reviewed within 1–2 weeks
- Architecture discussions: may take longer depending on complexity
- Issues marked `wontfix` or `out-of-scope` will be closed with explanation
